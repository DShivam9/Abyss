# AbsoluteUI Rules

# Core Philosophy

AbsoluteUI exists to create visually exceptional experiences.

Every decision should improve:

* Design
* Motion
* Performance
* Accessibility
* Developer Experience

---

# Rule 01 — Never Code Blindly

Never immediately start coding.

Always:

* Understand the objective
* Understand the desired experience
* Understand constraints
* Clarify ambiguity

Ask questions first when requirements are incomplete.

Discussion comes before implementation.

---

# Rule 02 — Architecture Before Code

Follow:

Research → Discussion → Planning → Implementation

Do not jump directly into development.

---

# Rule 03 — Skill Driven Development

Always leverage:

* Proven patterns
* Modern technologies
* Industry standards
* Performance best practices

The goal is not merely to make something work.

The goal is to build the best version possible.

---

# Rule 04 — Quality Over Speed

Never sacrifice quality for speed.

Prioritize:

* Better UX
* Better Architecture
* Better Performance
* Better Maintainability

---

# Rule 05 — Continuous Improvement

Every feature should be reviewed after completion.

Ask:

* Can visuals improve?
* Can motion improve?
* Can performance improve?
* Can accessibility improve?
* Can code improve?

The first version is rarely the final version.

---

# Rule 06 — Challenge Weak Decisions

Do not blindly agree.

If a proposed solution creates:

* Poor UX
* Technical debt
* Visual inconsistency
* Accessibility issues
* Performance problems

Provide alternatives.

---

# Rule 07 — Motion With Purpose

Motion must:

* Guide attention
* Create delight
* Improve clarity
* Increase immersion

Motion must never become decoration.

---

# Rule 08 — Beauty First

AbsoluteUI prioritizes beauty.

Every component should feel:

* Premium
* Memorable
* Intentional
* Polished

Generic components should not be shipped.

---

# Rule 09 — Quality Over Quantity

One exceptional component is more valuable than ten average ones.

Do not add components merely to increase library size.

---

# Rule 10 — No Trend Chasing

Avoid:

* Excessive glassmorphism
* Excessive glow
* Random gradients
* Design fads

Build timeless experiences.

---

# Rule 11 — Accessibility Required

Every component must support:

* Keyboard Navigation
* Focus States
* Semantic HTML
* Screen Readers

Accessibility is mandatory.

---

# Rule 12 — Production Ready Standard

Components must:

* Be Responsive
* Perform Smoothly
* Support Dark Mode
* Support Light Mode
* Scale Across Devices

No demo-only components.

---

# Rule 13 — Consistency First

All components must feel like they belong to the same design language.

If a component feels visually disconnected, redesign it.

---

# Rule 14 — Website Is The First Component

The AbsoluteUI website itself must showcase:

* Motion Quality
* Visual Quality
* Technical Quality

The website should be proof of the library's standards.

---

# Rule 15 — AbsoluteUI Standard

Before shipping, every component must answer YES:

* Is it beautiful?
* Is it memorable?
* Is it reusable?
* Is it performant?
* Is it accessible?
* Is it production ready?
* Is it better than the average alternative?

If the answer is NO to any question, it is not ready.

---

# Final Rule — Uniqueness & Curation

Never create a component simply because competitors have it.

Create a component only when AbsoluteUI can produce a noticeably better version.

Borrowing selected open-source components, patterns, and ideas from platforms like 21st.dev is allowed ONLY under proper licensing, with full attribution where required, and it must be elevated to meet AbsoluteUI quality standards. Never create a direct clone or repackage work without meaningful visual and technical improvement.
