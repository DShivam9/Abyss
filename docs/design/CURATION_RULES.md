# AbsoluteUI Curation Rules

## Purpose

This document defines the acceptance and rejection criteria for all components, animations, experiments, showcases, templates, and future contributions.

AbsoluteUI is a curated collection.

Not a component dump.

Not a component archive.

Not a collection of every possible UI pattern.

Every component must earn its place.

---

# Core Philosophy

The value of AbsoluteUI comes from curation.

Users should trust that:

> If a component exists in AbsoluteUI, it is worth studying, using, and bookmarking.

The goal is not to have the most components.

The goal is to have the best components.

---

# The AbsoluteUI Filter

Before any component is accepted, ask:

### Question 1

Is it visually memorable?

---

### Question 2

Does it provide a better experience than common alternatives?

---

### Question 3

Would a developer bookmark this component?

---

### Question 4

Would it still feel premium one year from now?

---

### Question 5

Does it represent the AbsoluteUI philosophy?

---

### Question 6

Does this component contribute to an immersive narrative or emotional experience?

If any answer is "No", reconsider inclusion.

---

# Acceptance Criteria

A component should satisfy all of the following:

## Visual Excellence

The component should feel:

* Crafted
* Premium
* Intentional
* Polished

Visual quality must be immediately apparent.

---

## Motion Quality

Motion must:

* Feel natural
* Feel smooth
* Enhance the experience

Motion should never exist purely for decoration.

---

## Technical Quality

The component must be:

* Responsive
* Accessible
* Performant
* Reusable

Beauty without quality is not enough.

---

## Reusability

A component should work across:

* Portfolios
* SaaS Websites
* Landing Pages
* Product Sites
* Agency Sites

Avoid niche implementations.

---

## Distinctiveness

The component should introduce:

* A unique interaction
* A refined visual treatment
* A better execution

Avoid redundant additions.

---

# Rejection Criteria

Immediately reject components that:

## Feel Generic

Examples:

* Standard Tailwind cards
* Basic buttons
* Generic hover effects

If the component can be found in thousands of libraries, it does not belong in AbsoluteUI.

---

## Copy Competitors Directly

Do not clone:

* Magic UI
* Aceternity
* Origin UI
* HyperUI
* shadcn
* 21st.dev

Learn from them.

Do not replicate them.

*Note on open-source borrowing:* Utilizing selected open-source components, patterns, and ideas from platforms like 21st.dev is permitted ONLY when licensing permits. You must follow original licenses, attribute authors where required, and meaningfully adapt and improve them to fit the AbsoluteUI high-taste standard. Direct repackaging or direct cloning is prohibited.

---

## Depend Entirely on Motion

If removing animation destroys the component:

Reject it.

The design must stand on its own.

---

## Follow Temporary Trends

Reject:

* Excessive glassmorphism
* Excessive glow
* Excessive blur
* Social-media-driven trends
* Dribbble gimmicks

Build timeless experiences.

---

## Introduce Visual Noise

Reject:

* Excessive movement
* Overwhelming effects
* Distracting visuals

Elegance beats spectacle.

---

## Create Accessibility Issues

Reject components that:

* Trap focus
* Ignore keyboard users
* Lack semantic structure
* Break screen reader support

Accessibility is mandatory.

---

# The 10x Rule

A component should not exist simply because competitors have it.

It should exist because AbsoluteUI can create a noticeably better version.

If the improvement is not obvious:

Do not build it.

---

# The Bookmark Test

Ask:

> Would a developer save this component for future use?

If the answer is uncertain:

Improve it.

If the answer is no:

Reject it.

---

# The Screenshot Test

Ask:

> If someone posted a screenshot of this component online, would people ask where it came from?

If not:

The component lacks distinction.

---

# The Hero Test

Ask:

> Is this component good enough to appear on the homepage?

If not:

Refine it.

Every component should be capable of becoming a featured component.

---

# Category-Specific Rules

## Buttons

Reject:

* Basic gradients
* Generic hover effects
* Simple scale animations

Accept:

* Memorable interactions
* Premium feedback systems
* Unique motion behavior

---

## Cards

Reject:

* Static layouts
* Generic product cards

Accept:

* Depth
* Motion
* Layered interactions

---

## Typography

Reject:

* Generic text reveals

Accept:

* Thoughtful storytelling
* Elegant motion
* Strong hierarchy

---

## Images

Reject:

* Basic zoom-on-hover

Accept:

* Immersive presentation
* Layered reveals
* Meaningful interaction

---

## Backgrounds

Reject:

* Resource-heavy effects
* Distracting visuals

Accept:

* Atmospheric depth
* Subtle enhancement

---

# Homepage Component Standard

Homepage components represent the highest quality tier.

Requirements:

* Exceptional visual quality
* Exceptional motion quality
* Strong originality
* Strong performance

Only the best work belongs on the homepage.

---

# Submission Review Process

Every component should pass:

## Stage 1

Experience Review

Questions:

* Does it contribute to an immersive narrative or emotional experience?
* Does it work within coordinated page compositions?
* Does it enhance scroll-driven storytelling?

---

## Stage 2

Concept Review

Questions:

* Is it useful?
* Is it unique?
* Does it align with AbsoluteUI?

---

## Stage 3

Design Review

Questions:

* Is it beautiful?
* Is it polished?
* Is it memorable?

---

## Stage 4

Motion Review

Questions:

* Is the motion intentional?
* Is the motion smooth?
* Is it necessary?

---

## Stage 5

Technical Review

Questions:

* Is it accessible?
* Is it responsive?
* Is it performant?

---

## Stage 6

Publication Review

Questions:

* Is it better than common alternatives?
* Is it worthy of AbsoluteUI?

Only then can it be published.

---

# Expansion Rules

Do not expand categories simply to increase component count.

Expansion should occur only when:

* Quality remains high
* Standards remain consistent
* New components introduce genuine value

---

# Long-Term Philosophy

The reputation of AbsoluteUI should come from trust.

Users should know:

* Every component is curated
* Every component is refined
* Every component has a reason to exist

The collection should remain small, intentional, and exceptional.

---

# Final Rule

A single unforgettable component is more valuable than one hundred forgettable components.

When in doubt, remove it.
